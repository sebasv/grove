# Grove v0.4 — Embedded Terminal (Single PTY per Worktree)

## Goal

Each worktree can host **one** live shell, rendered directly in grove's main pane. When the user hits `Enter` on a worktree row, grove spawns a shell with `cwd` = worktree path (if not already running) and shows it. Switching worktrees in the sidebar preserves the previous terminal — it keeps running in the background.

Multi-tab comes in v0.5. This version focuses on making the single-terminal case rock-solid.

## Acceptance criteria

- `Enter` on a worktree auto-spawns `$SHELL -l` with cwd = the worktree path (if no terminal exists for that worktree yet). The shell renders in the main pane.
- `Ctrl+Space` cycles focus between Sidebar and Main. A focus indicator (e.g. highlighted title or border colour) shows which pane owns input.
- When focus is **Main**, keys forward to the PTY:
  - printable chars, Enter, Backspace, Tab, Delete, Home/End, arrow keys, function keys → correct VT escape sequences.
  - Ctrl+<letter> → control bytes (0x01–0x1A).
  - `Ctrl+Space` is reserved for focus cycle (does not reach PTY).
- When focus is **Sidebar**, existing sidebar keys work as in v0.3.
- Main-pane resize propagates to PTY via `SIGWINCH` (handled by `portable_pty::MasterPty::resize`).
- Terminal output is coalesced at the UI layer: multiple `Event::TerminalOutput` messages between two frames cause one repaint.
- Terminals survive worktree switching; the last scroll state / cursor position is preserved.
- On quit, all PTY children are sent SIGHUP and the runtime exits cleanly.
- 67+ tests still pass; new tests for focus-cycle state machine and key-to-byte translation.

## Non-goals

- Scrollback mode (`Ctrl+\` / `Ctrl+[`) — postponed to v0.5.
- Multi-tab — v0.5.
- Mouse → PTY forwarding — v1.1.
- OSC/title updates from the shell — deferred; we can add in the polish pass.

## Data model

```rust
// model.rs
pub struct Worktree {
    pub branch: String,
    pub path: PathBuf,
    pub is_primary: bool,
    pub status: Option<WorktreeStatus>,
    pub terminal: Option<Terminal>,   // NEW

```

```rust
// terminal.rs
pub struct Terminal {
    pub parser: Arc<Mutex<vt100::Parser>>,
    writer: Box<dyn Write + Send>,
    master: Box<dyn MasterPty + Send>,
    _child: Box<dyn Child + Send>,    // dropped → child killed
}

impl Terminal {
    pub fn spawn(cwd: &Path, size: PtySize, wt_id: WorktreeId, tx: EventSender) -> Result<Self>;
    pub fn write(&mut self, bytes: &[u8]) -> io::Result<()>;
    pub fn resize(&mut self, size: PtySize) -> Result<()>;
    pub fn screen(&self) -> Arc<Mutex<vt100::Parser>> { self.parser.clone() }
}
```

### Reader thread

Each spawned terminal owns a dedicated OS thread that `read`s bytes from the PTY master and feeds them into `vt100::Parser`. After each successful read it sends `Event::TerminalOutput(WorktreeId)` to the async channel so the main loop knows to repaint.

Why an OS thread instead of `spawn_blocking`: repeatedly re-scheduling blocking tasks for a single long-lived read loop is wasteful. A thread is cheaper.

### Coalescing

On the main-loop side, after receiving at least one `TerminalOutput`, drain all additional `TerminalOutput` events from the channel without awaiting the UI. Only then draw one frame. This keeps the render rate capped near the display refresh (~60 Hz).

## Focus model

```rust
pub enum FocusZone {
    Sidebar,
    Main,
}

// UiState.focus_zone: FocusZone = Sidebar  // default
```

`Ctrl+Space` (reserved from **both** focus zones) toggles `focus_zone`. In Main focus, reserved keys stop at grove:

| Key | Handled by |
|-----|------------|
| `Ctrl+Space` | grove (cycle focus) |
| everything else | PTY |

`q` no longer quits when focus = Main (it's a character sent to the shell). Quit paths:
- Sidebar focus: `q` as before.
- Main focus: user must cycle back to Sidebar first, then `q`.

## Key → byte translation

Implemented as a small helper `key_to_pty_bytes(key: KeyEvent) -> Option<Vec<u8>>`:

| KeyCode | Bytes |
|---------|-------|
| `Char(c)` without Ctrl | UTF-8 encoding of `c` |
| `Char(c)` with Ctrl | `(c.to_ascii_uppercase() as u8) - 0x40` if `0x40..=0x5F`, else skip |
| `Enter` | `\r` (0x0D) |
| `Backspace` | `\x7f` (DEL) |
| `Tab` | `\t` |
| `Esc` | `\x1b` |
| `Up` | `\x1b[A` |
| `Down` | `\x1b[B` |
| `Right` | `\x1b[C` |
| `Left` | `\x1b[D` |
| `Home` | `\x1b[H` |
| `End` | `\x1b[F` |
| `PageUp` | `\x1b[5~` |
| `PageDown` | `\x1b[6~` |
| `Delete` | `\x1b[3~` |
| `F1..F4` | `\x1bOP`..`\x1bOS` |
| `F5..F12` | `\x1b[15~`..`\x1b[24~` |

## Rendering

When the active worktree has `Some(terminal)`, render the tui-term widget:

```rust
use tui_term::widget::PseudoTerminal;
use vt100::Screen;

let screen = term.parser.lock().unwrap();
frame.render_widget(PseudoTerminal::new(screen.screen()), area);
```

Otherwise render the current placeholder ("Active worktree" / "Select a worktree").

The main pane gets a bordered block with the worktree title; when focus = Main, the border is highlighted.

## Resize

On every draw, grove knows the main-pane Rect. If it differs from the terminal's last-known size, call `terminal.resize({rows, cols})`. This triggers SIGWINCH to the child; also re-initialises the vt100 parser's screen dimensions.

## Critical review (self)

### What could go wrong

1. **portable-pty unsafe code.** portable-pty wraps `openpty(3)` and `posix_spawn`. Well-audited; risk is low. No mitigation needed.

2. **Child processes on quit.** If grove panics or is killed, our reader threads and child shells may outlive the app. Partial mitigation: dropping the PtyPair closes the FDs, which should make the child get SIGHUP on next read/write. Full mitigation would be tracking PIDs and explicitly sending SIGHUP from a Drop impl. Not done for v0.4; acceptable because `kill $(pidof grove)` is a common fallback. Revisit if users complain.

3. **Terminal state for a closed shell.** If the user types `exit` in the shell, the child dies. The parser keeps the last-rendered screen. Pressing more keys just writes to a closed PTY (silent error). We do NOT auto-respawn; instead `Enter` on the same worktree will respawn. Acceptable v0.4 behavior.

4. **SIGWINCH timing.** Resizing on every draw is wasteful. Optimise: only call resize when the new size differs from the stored last-known size.

5. **vt100 escape-sequence coverage.** `vt100` handles ~95% of real-world output. Edge cases (sixel graphics, kitty image protocol) won't render; users will see stray glyphs. Document as a known limitation.

6. **Input latency.** Each keystroke walks: crossterm → EventStream → tokio channel → update → write to PTY. Typical round trip should be sub-millisecond. If it's noticeably slow, we'd need to inline the write for Main-focus keys and skip the channel round-trip. Not anticipated.

7. **Multiple readers.** vt100::Parser is NOT thread-safe on its own; we wrap it in `Arc<Mutex<_>>`. The reader thread and render thread both lock. Contention should be trivial (parser updates are fast).

8. **Ctrl+C forwarding.** User expectation is that `Ctrl+C` kills the foreground program in the shell. We translate `Ctrl+C` → byte 0x03 and forward. The PTY delivers SIGINT. Works.

9. **Focus ambiguity when no terminal exists.** If the user cycles focus to Main on an empty main-pane (no worktree active, or active worktree has no terminal), key forwarding is a no-op except Ctrl+Space returning to Sidebar. Make this explicit: render a hint line.

10. **Resize race with pending output.** If a big chunk of output lands while we're resizing, vt100 may wrap at the old size. Acceptable — the next output will render correctly; users don't notice in practice.

11. **Testing without a real PTY.** Unit tests exercise the key-to-byte mapping and focus state machine. Snapshot tests pre-load a `vt100::Parser` with canned output (no real shell spawn) to verify rendering. Actual PTY lifecycle is smoke-tested manually.

### Alternative designs considered

- **Spawn the shell via tokio::process::Command.** Gives async stdout but **no PTY** — no prompt line editing, no `vim`, etc. Rejected; a PTY is non-negotiable for our use case.
- **Use `tmux` for the terminal (tmux-mode integration).** Rejected in earlier conversation. Keeping the PTY fully in-process.
- **Grep out escape sequences and translate them ourselves.** Rejected — that's literally what vt100 does, and it's better than anything we'd write.

## Testing strategy

- **Unit: key_to_pty_bytes.** One test per KeyCode branch. ASCII chars, arrows, function keys, Ctrl+A/C/D.
- **Unit: focus cycle.** `Ctrl+Space` toggles; persists across other key events; doesn't leak PTY input into sidebar.
- **Unit: Worktree.terminal lifecycle via a mock spawn** — define a `TerminalFactory` trait so tests can inject a fake terminal struct without touching `portable-pty`.
- **Snapshot: main pane showing a pre-loaded vt100 screen.**
- **Snapshot: main pane hint when focus=Main and no terminal.**

Manual smoke test: launch grove, hit Enter on a worktree, run `vim`, `htop`, `ls --color`, resize the window. No crashes, no corruption.

## Out of scope for v0.4

- Multi-terminal tabs per worktree (v0.5).
- Scrollback mode (v0.5).
- PTY persistence across grove restarts (v1.5 via tmux backing).
- Shell session history restoration (not planned).
