# Grove v0.2 — In-App Repo Management

## Goal

Replace the v0.1 terminal round-trip (`grove` exits with a "run `--init`" message) with a TUI-driven onboarding: `grove` always launches into the TUI, shows an empty sidebar with a hint, and `a` opens an add-repo modal that validates and persists to `config.toml`. Also add `R` to remove repos.

This is the first version where the user can meaningfully modify state from within the TUI. It sets the pattern for every later input-driven modal (new worktree in v0.9, search in later versions).

## Acceptance criteria

- `grove` with no `config.toml` launches the TUI directly; sidebar shows "No repos — press `a` to add one". No first-run exit path.
- `grove --init` and `grove --print-paths` keep working (tooling shortcuts).
- Pressing `a` opens an add-repo modal with a text input.
- Typing populates the input; `Backspace`/`Delete` edit; `←`/`→`/`Home`/`End` move the cursor.
- `Enter` validates the path:
  - Must be an existing directory.
  - Must be a git repository (`git -C <path> rev-parse --is-inside-work-tree` returns true).
  - Name must not collide with an existing repo.
  - On failure, the modal stays open and displays an inline error. No disk writes.
- On success: `config.toml` is created/rewritten atomically with the new entry appended; the new repo appears in the sidebar with its worktrees loaded; cursor moves to the new repo; modal closes.
- Pressing `R` on a repo row opens a confirmation modal. `y`/`Enter` confirms, `n`/`Esc` cancels. On confirm: `config.toml` is rewritten without that repo; UI state referencing the repo (cursor, active worktree, expanded map) is cleaned up.
- `Esc` cancels any modal with no side effects.
- Lightweight test coverage: unit tests for TextInput, Config save round-trip, AppState add/remove semantics; snapshot tests for empty sidebar + modal render states.

## Data model additions

```rust
// src/app.rs

pub enum Modal {
    Help,
    AddRepo(AddRepoModal),
    ConfirmRemoveRepo { repo_idx: usize },
}

pub struct AddRepoModal {
    pub input: TextInput,
    pub error: Option<String>,
}
```

```rust
// src/ui/text_input.rs

pub struct TextInput {
    buffer: String,          // UTF-8
    cursor: usize,           // byte offset — always on a char boundary
}
```

`TextInput` exposes a narrow API: `insert_char`, `backspace`, `delete`, `move_left`, `move_right`, `home`, `end`, `value`, `cursor_display_col` (byte offset → column index via `UnicodeWidthStr`). For v0.2 we accept ASCII-only is fine in practice; the byte/char plumbing is still correct so unicode input won't panic.

## AppMessage additions

```rust
pub enum AppMessage {
    // ... existing ...

    OpenAddRepo,
    OpenConfirmRemoveRepo,      // uses current cursor position
    InputChar(char),
    InputBackspace,
    InputDelete,
    InputCursorLeft,
    InputCursorRight,
    InputHome,
    InputEnd,
    SubmitModal,
    CancelModal,

```

Using granular messages rather than a single `ModalKey(KeyEvent)` keeps `AppState::update` pure and testable (no KeyEvent parsing in update logic).

## Input routing

```rust
fn key_to_message(key: KeyEvent, modal: Option<&Modal>) -> AppMessage {
    if key.kind != KeyEventKind::Press { return NoOp; }
    match modal {
        None => default_keys(key),
        Some(Modal::Help) => help_keys(key),
        Some(Modal::AddRepo(_)) => text_input_keys(key, true /* allow submit */),
        Some(Modal::ConfirmRemoveRepo { .. }) => confirm_keys(key),
    }
}
```

## Config save

`Config::save(path)` atomic-writes the full serialized config (temp + rename). No comment preservation — users should be aware that `config.toml` is both hand-editable AND rewritten by the app (add/remove repo paths will wipe comments). We document this behavior in the generated template so it's not surprising.

An alternative (preserve comments via a formatter-aware TOML library like `toml_edit`) is explicitly deferred. `toml_edit` adds compile time and API complexity for an edge case that few users will encounter in v0.2.

## Critical review (self)

### What could go wrong

1. **TextInput unicode.** We use byte offsets for the cursor. If a user pastes a multibyte char (emoji, accented letter), we must:
   - Advance the cursor by the char's byte length on insert.
   - On backspace, step back to the previous char boundary, not just subtract 1.
   - On left/right arrow, step by char, not byte.
   Failing any of these panics on `insert_str`/`remove` with "not a char boundary". Fix: use `char_indices` iteration; test with an emoji in the unit tests.

2. **Path canonicalization.** If the user types `~/dev/foo` or a relative path, what do we do?
   - Expand `~` ourselves (shellexpand crate or manual).
   - Canonicalize with `std::fs::canonicalize` to resolve symlinks and relatives.
   Expansion rules: `~/` → `$HOME/`. No `$VAR` expansion — that's a rabbit hole. Relative paths resolved against CWD (the user's terminal CWD at launch, not the worktree path).

3. **Name collision.** Two repos can't share a name (state.toml keys by name). We use `Path::file_name()` as the default name. If there's a collision, we should either:
   - Prompt for a custom name (modal gets more complex).
   - Append a disambiguator (`name-2`).
   For v0.2: fail with "a repo named 'X' already exists". User then manually chooses a different parent dir or edits config to rename. Future work: second modal field for name.

4. **Config file rewrite wipes comments.** Users who hand-edit `config.toml` and add comments will lose them the next time they add/remove a repo via the TUI. Mitigation: top-of-file comment in the template warns about this. Consider `toml_edit` later if anyone complains.

5. **Atomic write crash safety on network drives.** `rename` isn't strictly atomic across filesystems. Config dir is normally local; this is academic. Not worth addressing.

6. **Race conditions.** If the user edits `config.toml` in a text editor while grove is running and then adds a repo in the TUI, the TUI's in-memory config overwrites the file. Could add a mtime check on save and warn. Deferred — clearly documented as "grove owns the file while running" is enough.

7. **Sidebar cursor after remove.** If the removed repo was under the cursor, cursor needs to move. To previous repo, next repo, or None (if it was the last). Simple policy: move to the same index (which now points at the next repo), or to `None` if empty. Edge: cursor was on a worktree of the removed repo — same rule at the repo level.

8. **Active worktree referencing the removed repo.** Must clear `active_worktree` if it pointed into the removed repo.

9. **First-run cursor default.** With no repos, cursor is `None`. Sidebar render must not crash when asked to draw the cursor on an empty tree. The existing v0.1 code already handles this because cursor is Option; verify.

10. **`R` ambiguity.** `R` (capital) was already mapped to "remove repository" in the v0.1 sidebar keybindings table (listed but disabled). We're activating it. Lowercase `r` is "refresh git status" planned for v0.3. Capitalization distinguishes destructive from safe action. Acceptable, matches vim idiom.

11. **Modal keys must not leak.** While a modal is open, `j/k/h/l/a/R/?` must not reach the sidebar. Already true in v0.1 for Help modal; need to keep the discipline.

### Alternative designs considered

- **Single big "Manage repos" modal** (list of repos with add/remove actions inside). Rejected — modal-within-modal feels heavy. Stick with per-action modals.
- **Auto-create config on first launch with an empty `[[repos]]` section.** Rejected already in the prior conversation (silent file creation in `~/Library/...`). Instead we only write `config.toml` on the first successful add, which is explicit user consent.
- **Let `a` open an OS file picker instead of a text input.** TUI purity argues against spawning a GUI. A fuzzy finder for git dirs under `$HOME` could be a v1.x polish item.

### What this introduces as debt

- Config rewrites lose comments. Tracked as a known limitation, fix with `toml_edit` if needed later.
- No multi-field forms (just a single text input). The modal framework will need generalization in v0.9 (new worktree modal: branch name + base branch + auto-push checkbox).
- TextInput is single-line. Later phases that want paragraph input (commit message editing?) will need a second variant.

### Testing strategy

- **Unit tests**
  - `TextInput`: insert ASCII, insert emoji (4-byte UTF-8), backspace across char boundary, move_left/right stay on boundaries, home/end.
  - `Config::save`: save then reload equals original.
  - `AppState::add_repo`: with valid fixture path (a temp git repo created during test) adds to both config and repos.
  - `AppState::add_repo`: with non-existent path returns error, state unchanged.
  - `AppState::add_repo`: with non-git path returns error, state unchanged.
  - `AppState::add_repo`: with duplicate name returns error, state unchanged.
  - `AppState::remove_repo`: clears matching active_worktree; moves cursor sensibly.
- **Snapshot tests**
  - Empty sidebar.
  - AddRepo modal: empty input.
  - AddRepo modal: with typed text.
  - AddRepo modal: with error message.
  - ConfirmRemoveRepo modal.

## Out of scope for v0.2

- Editing an existing repo's name or base_branch in-app (hand-edit `config.toml`).
- Repo reordering.
- Search/fuzzy add (pick from a list of git dirs under `$HOME`).
- Recursive worktree discovery (still shells out to `git worktree list --porcelain` — switching to `git2` is v0.3).
