# Grove v1.0 — Polish, Themes, 1.0.0

## Goal

Consolidate v0.1–v0.9 into a stable 1.0 release. Minimal new features: a theme switcher, a polished help overlay listing all keybindings, and an explicit statement that keybindings are frozen for the 1.x series.

## Scope

1. **Theme support.** `[theme] base = "default" | "tokyonight" | "gruvbox"` in `config.toml`. Controls accent (focus borders, cursor highlight), dim, and key status colors.
2. **Keybinding stability.** 1.0 freezes the keymap. v1.2 will add user-configurability but 1.x upgrades never silently move a binding.
3. **Version bump to 1.0.0.**

## Non-goals

- Custom theme files (deferred; themes are a closed set for 1.x).
- New keybindings.

## Implementation

```rust
// src/theme.rs
pub struct Theme {
    pub accent: Color,
    pub dim: Color,
    pub success: Color,
    pub warn: Color,
    pub danger: Color,
}

pub fn load(name: &str) -> Theme { ... }
```

Config grows `[theme] base = "default"` with a serde default. Widgets that currently hard-code `Color::Cyan` or similar pull from `AppState.theme` instead.

Theme list for 1.0:
- `default` — ratatui defaults (grove v0.x behaviour).
- `tokyonight` — blue accent, yellow warn, magenta danger.
- `gruvbox` — orange accent, aqua success.

## Critical review (self)

1. **Invasive color changes.** Hard-coding is scattered; migrating every widget to read from `theme` is tedious. Mitigation: only migrate the widgets most affected (sidebar cursor, main-pane border, badge colors). Accept that some fixed colors remain for 1.0; refactor in a later polish pass.

2. **Unknown theme name in config.** Fall back to `default` with a warning printed before the TUI starts.

3. **Accessibility.** None of our built-in palettes are tuned for low-vision users. Worth documenting but not blocking.
