# Grove v0.5 — Terminal Tabs + Scrollback

## Goal

Extend the single-terminal v0.4 to multiple terminals per worktree (tab bar above the main pane) and add scrollback-mode navigation of the vt100 parser's history.

## Acceptance criteria

- Each worktree can host N ≥ 0 terminals; `AppState.terminals: HashMap<WorktreeId, WorktreeTerminals>` where `WorktreeTerminals { list: Vec<Terminal>, active: usize }`.
- Tab bar renders above the main pane: `[▶ 1] [2] [3] [+]`. Active tab highlighted; `+` suggests where new-tab lives (visual only).
- Keybindings (focus=Main):
  - `Ctrl+T` → spawn new terminal for the active worktree, make it active.
  - `Ctrl+W` → kill the active terminal; closes that tab. If it was the last, drop WorktreeTerminals; fall back to placeholder.
  - `Ctrl+Right` / `Ctrl+Left` → cycle active tab.
- Scrollback mode:
  - `Ctrl+\` toggles between Insert and Scrollback. Visible indicator in the tab title (e.g., `[1 ⇅]`).
  - In Scrollback: `k`/`Up` scrolls up one line, `j`/`Down` down, `PgUp`/`PgDn` page, `g`/`G` top/bottom, `Esc` or `i` returns to Insert.
  - vt100 scrollback is exposed via `Parser::screen().scrollback()` API (read-only view offset). We shift the rendered screen by that offset.
- Single-terminal path still works (Enter on a new worktree spawns the first tab).
- 76+ tests still pass; new tests for tab state transitions and scrollback toggle.

## Non-goals

- Per-tab names / titles (derived from OSC-1 in a later polish pass).
- Search within scrollback (v1.x).
- Split panes within a worktree (out of scope for the foreseeable future).

## Data model

```rust
// app.rs
pub struct WorktreeTerminals {
    pub list: Vec<Terminal>,
    pub active: usize,
    pub mode: TerminalMode,  // Insert | Scrollback
    pub scroll_offset: usize,
}

#[derive(Clone, Copy, PartialEq, Eq)]
pub enum TerminalMode { Insert, Scrollback }
```

## Critical review (self)

1. **Ctrl+\ vs Esc (= Ctrl+[).** `Ctrl+[` collides with Escape which shells and vim expect to pass through unchanged. Using `Ctrl+\` sidesteps the collision. `Ctrl+\` also sends SIGQUIT by default in real shells — but since we intercept it at grove, the shell never sees it. Document clearly.

2. **Killing active tab with Ctrl+W.** `Ctrl+W` is "word delete" in most readline shells. Hijacking it prevents the user from deleting a word in their shell prompt. Mitigation: Ctrl+W only triggers "kill tab" in Scrollback mode, OR require a prefix. For simplicity in v0.5, remap to a prefix-free key: `Ctrl+D` when the shell is idle? No — that's shell EOF. Let me use a double-press: `Ctrl+W Ctrl+W` (grove command mode). Too fancy. Actually simplest — require scrollback mode for destructive actions. In Insert mode, Ctrl+W passes through (word delete); in Scrollback mode, Ctrl+W closes the tab.

3. **Scroll offset persistence.** When user switches tabs or worktrees, does scrollback position reset? Design says it's per WorktreeTerminals, so yes it persists per worktree. Switching back to a scrolled-up worktree preserves view.

4. **Tab width on narrow terminal.** If a worktree has 8 tabs, "[▶ 1][2][3][4][5][6][7][8][+]" may overflow. Truncate with "…" overflow, or page. For v0.5 we just let the tab bar clip.

5. **Re-indexing terminals after tab close.** If active=2 and we close tab 1, active should become 1 (the old tab 2, now at index 1). Straightforward arithmetic but easy to get wrong — covered by unit tests.

6. **Exiting shells while focus is on a different tab.** Current behavior (v0.4): shell dies silently, vt100 keeps last screen. Same for tabs. No auto-close on exit. User presses Ctrl+W to close.
