# Grove v1.3 — Onboarding polish, unified worktree modals, non-blocking UX

## Goal

Three threads, tied together by one principle: **the UI never waits on
the network or the filesystem.** Every operation that could take more
than a frame returns control to the user immediately, runs in the
background, and surfaces progress visibly.

- **Onboarding.** One command (`grove [<path>]`) that does the right
  thing whether you point it at a single repo, a parent directory full
  of repos, or nothing at all.
- **Worktree modal consolidation.** Creation and deletion each collapse
  to a single decision point instead of a mode toggle or cascading
  confirms.
- **Activity system.** Grove periodically fetches in the background so
  branch lists and base branches are already fresh when the user needs
  them. A footer row shows what's currently running.

v1.2 (configurable keybindings, PR #11) is orthogonal and progresses on
its own branch. This release slots between v1.1 and that work.

## Non-blocking-UX contract

A single rule drives the rest of the design:

> Pressing a key must never block on I/O. The slowest path from keypress
> to next frame is local state mutation.

Concretely:

- No synchronous `git fetch`, `git clone`, `gh api` calls in response
  to a keypress.
- Modal opens render immediately from cached state. If cached state is
  stale, the modal shows `⟳` next to affected items and re-renders when
  the background task finishes.
- Actions that create worktrees or branches run against the freshest
  cached refs. The background sync loop is responsible for "fresh
  enough"; the modal is not.
- When an action does need a one-shot network hit (rare), it runs
  async and the modal shows the spinner until the result arrives. The
  user can still `Esc` to cancel.
- A persistent activity footer at the bottom of the sidebar lists any
  currently-running background task (fetch, PR poll, status refresh)
  so the user understands why a badge just changed or a list just
  reordered.

## Non-goals

- Configurable keybindings (v1.2, separate branch).
- Multi-remote selection — we track `origin` only. Users with multiple
  remotes still pick a remote branch via the displayed
  `<remote>/<branch>` entry.
- Fetching across all repos at TUI startup in a blocking burst. Fetches
  are staggered by the activity scheduler.

### PR 2 — Modal input field styling

**Problem.** Text inputs inside modals (the AddRepo path field, the
NewWorktree branch-name field) are rendered with the same background
as the rest of the modal body. A user opening the modal for the first
time has to read hint text to understand where their keystrokes will
land. For a TUI that leans heavily on modal input, this is a tiny but
persistent friction.

**Proposal.** Add an `input_bg` (and derived `input_bg_focused`) colour
to the `Theme` struct. The shared `text_input::render` widget draws
its content area with that background. All three built-in themes get
a sensible default:

- `default`: `Color::Rgb(32, 34, 42)` — a shade darker than a dark
  modal, matching ratatui defaults.
- `tokyonight`: `Color::Rgb(30, 33, 48)`.
- `gruvbox`: `Color::Rgb(40, 40, 40)`.

The unfocused-input colour is slightly dimmed (the NewWorktree
`PickBranch` mode has the input unfocused when the user is arrowing
through branches). When the unified new-worktree modal (PR 6) lands,
"input focused" is always true, so the dim variant is temporary
scaffolding.

**State changes.**
- Extend `src/theme.rs` `Theme` with `input_bg: Color` and
  `input_bg_focused: Color`. Update the three theme constructors.
- `src/ui/text_input.rs` `render` sets the `Style` background on the
  rendered span to `theme.input_bg_focused` (or `input_bg` when
  `focused: bool` is false — widget signature grows a `focused`
  parameter).
- Callers (`ui/add_repo.rs`) pass the appropriate flag.

**Acceptance.**
- Open the AddRepo modal: the input row is visually distinct from
  the rest of the modal body.
- Open the NewWorktree modal in `NewBranch` mode: same.
- In `PickBranch` mode the input row is present but dimmed,
  signalling "you're not typing here right now."
- Snapshot tests for add_repo and new_worktree capture the new
  colours; insta review accepted.

### PR 3 — Activity system & background sync

**Why it comes first.** Every later UX improvement in this release
leans on "work runs asynchronously, progress is visible." Building
that machinery as a single focused PR avoids duplicating the pattern
in three downstream PRs.

**Activity state.**

```rust
pub struct Activity {
    pub id: ActivityId,             // monotonic counter
    pub kind: ActivityKind,         // Fetch | PrPoll | StatusRefresh | Scan
    pub scope: ActivityScope,       // Repo(idx) | Global
    pub started_at: Instant,
    pub label: String,              // "fetching origin (grove)"
}

pub struct ActivityState {
    pub in_flight: Vec<Activity>,
}
```

Tasks register themselves on spawn and deregister on completion. The
renderer reads `in_flight` to draw the footer.

**Footer rendering.** Bottom row of the sidebar (above the existing
divider line). Examples:

- idle: blank
- one task: ` ⟳ fetching origin/grove`
- multiple: ` ⟳ 3 background tasks` (dim; `?` menu lists them)

Auth warning (added in PR 5) shares this row: if idle and
unauthenticated, show `⚠ PRs: not authenticated`.

**Background fetch scheduler.**

- On grove start, spawn a `FetchScheduler` tokio task.
- Each repo has a fetch cadence (default: 300s). After `now - last ≥
  cadence`, schedule a fetch. Stagger repos by `repo_idx *
  cadence/N` so fetches don't all fire at once.
- Debounce: never run two fetches for the same repo concurrently.
- Events: `FetchStarted(repo_idx)` → registers Activity;
  `FetchFinished(repo_idx, Result)` → deregisters, updates
  `last_fetched_at`, triggers a status + branch-list refresh.
- Errors (network down, auth) are logged to the grove log, not
  surfaced in the UI except as "last fetch failed" dimness if we get
  around to it. A repeated failure doesn't spam the log — exponential
  backoff on the cadence up to 30 min.

**User-triggered fetch.** A new action (unbind) or via `r` (currently
"refresh all statuses"): broadens `r` to also kick off an immediate
fetch for all configured repos. This is opt-in per keypress, not a
block — the UI never stalls.

**Staleness surface (minimal for v1.3).** Each worktree's status
badges already reflect ahead/behind. No new staleness indicator for
the sidebar in this release. The activity footer is enough signal
that "things are updating."

**State / config additions.**
- `ActivityState` on `AppState`.
- `last_fetched_at: Option<Instant>` per repo (in-memory; not
  persisted).
- `fetch_cadence_secs: Option<u64>` in `[general]` config, default 300.

**Acceptance.**
- Launch grove. Activity footer shows `⟳ fetching…` briefly on
  startup, then clears.
- Leave grove idle for 5+ minutes. Fetches fire in the background; the
  activity footer blips; branch lists (when opened) reflect any new
  remote refs without the user explicitly refreshing.
- Pull the network plug. Fetches fail quietly; UI stays responsive.

### PR 4 — Unified entry: `grove [<path>]`

Replaces the previous plan's split between `grove [<path>]` and
`grove scan <dir>`. One command covers both shapes.

```
grove                   # open TUI with existing config
grove .                 # treat CWD as the target
grove ~/dev/my-repo     # target is a single repo
grove ~/dev             # target contains multiple repos
```

**Algorithm.**

1. Resolve `path` to an absolute dir. If it doesn't exist, error out.
2. If `path` itself is a git repo (has `.git` or is a bare repo with
   worktrees), treat as **single-repo mode**:
   - If already in config, open the TUI on its primary worktree.
   - If not, add to config (validating + deriving base branch via the
     same logic as the empty-state "add" modal), save, open on it.
3. Otherwise, walk `path` one level deep by default (configurable via
   `--depth`, max 3) and collect every directory that looks like a
   git repo. This is **scan mode**:
   - Found zero: print a friendly "no git repos found under <path>"
     and exit nonzero.
   - Found any: open TUI with a modal listing the discovered repos,
     each toggleable on/off (all-on by default, duplicates of
     already-configured repos dimmed and off by default). Confirming
     adds the selected repos to config and populates the sidebar.
4. No positional arg → existing behaviour (open TUI with current
   config, empty-state modal if none).

**Why TUI-confirmation instead of non-interactive adds.** Scanning can
easily pick up dozens of repos (work + dotfiles + side projects).
Running `grove ~/` would be a footgun. The discovery modal makes the
set visible and editable before it hits disk. `--yes` / `-y` is still
a flag for scripting.

**State changes.**
- New `Modal::DiscoveredRepos { candidates: Vec<DiscoveredRepo> }`
  where each candidate carries `path`, `name`, `already_configured`,
  `selected`.
- CLI: `path: Option<PathBuf>` positional on `Cli`. `depth: u8` behind
  `--depth`.
- Git-walk helper that respects `.gitignore` of the parent directory
  (don't recurse into `node_modules`) and skips hidden dirs except
  `.git`.

**Feedback:** discovery itself can be slow on deep trees. Spawn it as
an `Activity::Scan`; modal shows `⟳ scanning…` until `ScanDone` fires.

### PR 5 — Persistence & auth surfaces

**F2 theme persists.** `AppMessage::CycleTheme` writes
`config.theme.base` back through `Config::save`. Errors go to the log,
not fatal. Persist on every cycle (config is tiny; no debounce
needed).

**Base-branch auto-detect.** In `try_add_repo`, after repo validation:

1. Check for `origin/HEAD` via libgit2.
2. If absent, ask the scheduler to fetch this repo now
   (`ActivityKind::Fetch`, promoted priority), and resolve
   `origin/HEAD` once the fetch finishes. This runs in the background;
   the repo is added immediately with `base_branch = None` in config
   so the fallback to `general.default_base_branch` applies, then
   patched with the real value once detection completes.
3. If no remote is configured, leave `base_branch` unset.

This keeps repo-add instant even on a slow remote, while still
auto-detecting the right base when the network cooperates.

**GitHub auth indicator in activity footer.** When no fetch/poll is
running, if at least one configured repo has a GitHub remote and
`github::build_client()` returned `None`, show:

```
⚠ PRs: not authenticated — ? for setup
```

The `?` overlay keeps its current GitHub-auth line so the hint is
actionable.

### PR 6 — Unified new-worktree modal

**Current state.** Two explicit modes (`PickBranch` / `NewBranch`)
toggled by `Tab`, each with its own rendering. User has to know which
mode they're in.

**Proposed.** One modal, one input field, filterable list below.

```
┌─ New worktree in grove ──────────────────────────────┐
│                                                       │
│  Search / new branch name: feat/search█               │
│                                                       │
│  ▶ Create new branch  feat/search                     │
│    feat/sidebar                                       │
│    fix/deps                                           │
│    origin/feat/remote-only         [remote]           │
│                                                       │
│  ⟳ fetching origin (grove)                            │
│  j/k navigate · Enter confirm · Esc cancel            │
└───────────────────────────────────────────────────────┘
```

- First row is always "Create new branch `<input>`" (disabled only
  when input is empty or invalid as a git ref).
- Below: existing branches (local + remote, deduplicated),
  substring-filtered by input text.
- Arrow keys / `j` / `k` move the cursor. Enter on the "create new"
  item creates a new branch off the repo's base; Enter on an existing
  branch checks it out (or tracks a remote-only branch).
- Typing updates both the "create new" line and the filtered list.

**Interaction with the activity system.**

- On open, list renders from cache immediately. If
  `last_fetched_at` for the repo is older than
  `fetch_cadence_secs / 2`, nudge the scheduler to fetch now
  (priority boost, still async). The modal shows `⟳ fetching…` in
  its footer while the task is in-flight; when it completes, the
  branch list re-renders.
- Pressing Enter never blocks. The worktree is created off the
  current cached HEAD of the base branch (which, thanks to the
  scheduler, is usually fresh within 5 min). If the branch being
  created off a base that hasn't been fetched *ever* (e.g.,
  brand-new repo, offline), we create off the local tip and add a
  dim status line in the worktree row: `(base: local only)`.
- `r` inside the modal forces an immediate fetch.

**State changes.**
- Drop `NewWorktreeMode` enum.
- `NewWorktreeModal` gains `filter: String`, `filter_matches:
  Vec<BranchEntry>`, `fetch_pending: bool`.
- `git::fetch_remote(repo_root)` new helper, spawned via
  `async_evt::spawn_fetch` → `Event::FetchFinished(repo_idx,
  Result)`.

### PR 7 — Unified delete-worktree modal

**Current state.** Confirming worktree removal chains to a second
modal about branch deletion, and a third about force-delete when the
branch has unmerged commits. Three keystrokes minimum, often four,
and the third modal is a nasty surprise.

**Proposed.** Single modal with exactly two options — keep, or a
single contextual delete. The delete variant is chosen at modal-open
time based on whether the branch is fully merged into its base:

- merged → regular delete (`git branch -d`), bound to `d`
- unmerged → force delete (`git branch -D`), bound to `D`

The `d`/`D` distinction is deliberate: capital-D is harder to hit by
accident, and forces the user to acknowledge that they're destroying
unmerged work. The two keys are also *absolute* — pressing `d` on
an unmerged branch does nothing (with an inline reminder "use `D` to
force-delete"), and `D` on a merged branch isn't offered (redundant).
This sidesteps arrow-navigate + Enter, where it's easy to pick the
wrong thing by muscle memory.

Merged case:

```
┌─ Remove worktree for feat/sidebar ───────────────────┐
│                                                       │
│  Worktree path: ~/dev/grove-feat-sidebar              │
│                                                       │
│  [k] Remove worktree, keep branch  (default)          │
│  [d] Remove worktree and delete branch (-d)           │
│                                                       │
│  k keep · d delete · Esc cancel                       │
└───────────────────────────────────────────────────────┘
```

Unmerged case:

```
┌─ Remove worktree for feat/sidebar ───────────────────┐
│                                                       │
│  Worktree path: ~/dev/grove-feat-sidebar              │
│                                                       │
│  ⚠ "feat/sidebar" has unmerged commits                │
│  ⚠ PR #42 is open — force-delete will close it        │
│                                                       │
│  [k] Remove worktree, keep branch  (default)          │
│  [D] Remove worktree and force-delete branch (-D)     │
│                                                       │
│  k keep · D force-delete · Esc cancel                 │
└───────────────────────────────────────────────────────┘
```

- `k` or `Enter` → keep branch (safest).
- `d` → regular delete (only in merged case). In unmerged case shows
  an inline "use `D` to force-delete" hint and stays open.
- `D` → force delete (only offered in unmerged case, but accepted
  silently in merged case since it's a no-op difference from `d`).
- `Esc` → cancel.
- No arrow navigation; absolute keys verify intent.

**Merged-check implementation.** `git::is_branch_merged(repo_root,
branch, base)` = returns true iff every commit in `branch` is
reachable from `base`. Internally `git merge-base --is-ancestor
<branch> <base>` or the libgit2 equivalent. Fast (no network).

**PR-open warning** renders in the unmerged case above the options
(since the destructive action there is force-delete, which closes the
PR). In the merged case the regular `-d` is safe, so no PR warning
even if one is open.

All options complete fast (local-only); no activity-system coupling.

**State changes.**
- Drop `Modal::ConfirmDeleteBranch` and `Modal::ForceDeleteBranch`
  variants entirely — the cascade disappears.
- `Modal::ConfirmRemoveWorktree` gains `variant: DeleteVariant`
  (`Merged` | `Unmerged`), populated by `is_branch_merged` when the
  modal opens.
- Key handler for this modal: `k`/Enter → keep, `d` → regular delete
  (merged variant only, hint otherwise), `D` → force delete (accepted
  always but only displayed in unmerged variant), `Esc` → cancel.
- Submit handler runs the selected combination; inline error keeps
  the modal open on partial failure.

## Critical review (self)

1. **Activity system is the biggest piece.** PR 3 is load-bearing for
   PR 6 and (partially) PR 5. If reviewers want to split it further,
   the fetch scheduler and the activity footer can land as two
   sub-PRs. Resist unless review asks — smaller slices make the end
   state harder to reason about.

2. **Background fetches on slow networks.** Fetch cadence of 5 min is
   fine on a fast connection. On tethered / overseas / corporate
   proxies it could be painful (battery, data). Mitigations: (a)
   exponential backoff on failure (30 min cap); (b) honour
   `GIT_HTTP_LOW_SPEED_LIMIT` by shelling out to `git fetch` rather
   than using libgit2 (so user's git config drives timeouts); (c)
   expose `fetch_cadence_secs` in config so users can set it to
   something conservative or zero (disabled).

3. **`grove ~/` is still a footgun even with a confirmation modal.**
   Modal lets you deselect, but a paste of 40 rows is annoying. The
   default depth of 1 plus the explicit modal is good enough. Users
   who want `~/dev/*` should pass that.

4. **Activity footer vs terminal's own status line.** When the main
   pane is a terminal, the activity footer lives in the sidebar area,
   so no collision. Confirmed.

5. **Enter in the new-worktree modal and "branch off local only."**
   When the base branch has never been fetched, we fall back to local
   tip. That's a silent behaviour divergence ("I created feat/x off
   main, why is it old?"). The dim `(base: local only)` row addresses
   this, but only if the user reads the sidebar row. Alternative:
   render a one-line banner above the worktree on first view of the
   new worktree. Defer to feedback.

6. **Unified delete modal — `y`/`n` compat.** Previous flow mapped
   `y`/`n` to submit/cancel. The new modal uses `k` / `d` / `D` /
   `Esc`. Users with `y`-muscle-memory will get a no-op instead of a
   destructive action — acceptable; we should not re-bind `y` to any
   of the three options since it's ambiguous which one.

7. **Base-branch auto-detect relies on a fetched remote.** We now fix
   that by auto-fetching in the background as part of repo-add. Until
   that completes, config says `base_branch = None` and we fall back
   to `general.default_base_branch`. Once detect finishes the config
   is patched. User never sees a wrong base diff, just a generic one
   if they're fast enough.

8. **`--init` removal is a breaking CLI change.** Users scripting
   `grove --init` in a dotfiles setup will see an error. Acceptable
   given v0.2 shipped the replacement; note in release notes.

9. **Scheduler running in the background on a cloned machine.** If
   the machine has no keychain access for SSH keys, every fetch fails.
   That would be noisy without the backoff; with the backoff it's
   silent. Good enough for v1.3; revisit if users complain.

10. **Scope creep.** The design keeps growing because each pushback
    adds infrastructure (activity state, scheduler, footer). That's
    the right trade for the "never wait" contract — anything smaller
    pushes latency back onto the user. Accept the scope increase;
    don't pretend this is a small release.
