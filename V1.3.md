# Grove v1.3 — Onboarding polish & unified worktree modals

## Goal

Reduce friction between "I installed grove" and "grove is useful," and
collapse the multi-step worktree create/delete flows into single decision
points.

Two independent tracks:

- **Onboarding & defaults.** README mismatch, missing CWD entry point,
  silent GitHub auth, non-persistent theme, base-branch defaulted to `main`
  even when the repo uses `master`.
- **Worktree modal consolidation.** Creation and deletion both spread
  across multiple modes / cascading confirmation modals. Each should be a
  single modal with all options visible.

v1.2 (configurable keybindings, PR #11) is orthogonal and progresses on
its own branch. This release slots between v1.1 and that work.

| Auto-fetch base branch before branching off it | new |

## Non-goals

- Configurable keybindings (v1.2, separate branch).
- Multi-remote selection in the new-worktree modal — we track
  `origin` only. Users with multiple remotes still pick a remote branch
  via the displayed `<remote>/<branch>` entry.
- Fetch-all-repos-on-startup; fetch is scoped to the repo the user is
  acting on.

## Details

### PR 1 — Docs & CLI cleanup

- **README Quickstart.** Replace the "create a TOML file" section with:
  ```
  brew install grove
  grove
  # press `a` to add your first repository
  ```
  Keep the TOML example later in the doc as a reference for power users.
- **`--init` removal.** Empty-state modal (v0.2) makes it dead surface.
  Delete from `Cli` in `src/main.rs`, delete `Config::write_template` /
  `TEMPLATE` in `src/config.rs`, drop its test.
- **`--print-paths` doc.** Append a one-liner to the README: "Run
  `grove --print-paths` to see config / state / log locations."
- **Keybind quirks.**
  - `q` quits from any focus when no modal is open (currently sidebar
    only).
  - `?` opens help from any focus (currently sidebar only).
  - `Ctrl+C` with no modal quits (currently reaches PTY, never quits
    grove).
  - Update help overlay (`src/ui/help.rs`) accordingly.
- **Help overlay hint for GitHub auth** stays where it is.

Keybinds are still listed as "frozen for 1.x" (v1.0 promise) — these are
additions, not rebinds.

### PR 2 — One-shot CWD & bulk scan

**`grove [<path>]`**
- Positional optional arg. When given, expand to absolute path and
  verify it's a git repo; error out otherwise.
- If `path` matches an existing repo in config (by absolute path), open
  on that repo's primary worktree.
- If not in config, add it to `config.toml` (same validation as
  `try_add_repo`), save, then open on it.
- `grove .` resolves the current directory. If CWD is inside a worktree
  of a known repo, open on that worktree.
- CWD matching walks up directories until it finds a git repo root; if
  none found, falls back to normal "open sidebar" behaviour with a
  one-line note.

**`grove scan <dir>`**
- Walks `<dir>` one level deep (configurable with `--depth N`, default 1)
  looking for `.git` directories.
- Prints a list, asks for confirmation (y/N) unless `--yes` is passed.
- Adds each to config, deriving `name` from directory name. Skips
  duplicates silently.
- Non-TUI command; exits after adding. Running `grove` afterwards shows
  the populated sidebar.

### PR 3 — Persistence & auth surfaces

**F2 theme persists.** `AppMessage::CycleTheme` writes the new
`config.theme.base` back through `Config::save`. Errors are logged, not
fatal.

**Base-branch auto-detect.** In `try_add_repo`, after validating the
repo, run `git symbolic-ref refs/remotes/origin/HEAD` to resolve the
default branch. On success, store as `repo.base_branch` in the config.
On failure (no remote, unfetched), fall back to the existing
`default_base_branch`.

**GitHub auth indicator.** Sidebar footer (new one-line row at bottom of
sidebar):

- authenticated: nothing shown (or a subtle dim check)
- unauthenticated with repos that have a GitHub origin: `⚠ PRs: not
  authenticated — ? for setup`

Shown only when at least one repo has a GitHub remote. `github.rs`
already has `discover_owner_repo` we can reuse for detection.

### PR 4 — Unified new-worktree modal + auto-fetch

**Current state.** Two explicit modes (`PickBranch` / `NewBranch`)
toggled by Tab, each with its own rendering. User has to know which
mode they're in.

**Proposed.** One modal, one input field, one list below.

```
┌─ New worktree in grove ──────────────────────────────┐
│                                                       │
│  Search / new branch name: feat/search█               │
│                                                       │
│  ▶ Create new branch  feat/search                     │
│    feat/sidebar                                       │
│    fix/deps                                           │
│    origin/feat/remote-only         [remote]           │
│                                                       │
│  j/k navigate · Enter confirm · Esc cancel            │
└───────────────────────────────────────────────────────┘
```

- First item is always "Create new branch `<input>`" (disabled only when
  input is empty or invalid as a git ref).
- Remaining items = existing branches (local + remote, deduplicated),
  fuzzy-filtered by the input text.
- Arrow keys / `j` / `k` move the cursor. Enter on the "create new"
  item creates a new branch; Enter on an existing branch creates a
  worktree checking that branch out (or tracks a remote-only branch).
- Typing updates both the "create new" line and the filtered list; the
  cursor stays on the first item by default.

**Auto-fetch.**
- When the modal opens, spawn a background `git fetch --all --prune`
  for the repo. List renders immediately from cached state; if the
  fetch yields new refs the list re-renders.
- When the user confirms a "create new branch" off the base branch,
  fetch that branch's upstream synchronously before creating the
  worktree, so `feat/x` branches off a freshly-pulled `origin/main`.
  Don't sync-fetch for existing-branch paths — the user explicitly
  picked what they wanted.
- Render a small `⟳ fetching…` hint in the modal while a fetch is
  running.

**State changes.**
- Drop `NewWorktreeMode` enum.
- `NewWorktreeModal` gains `filter_matches: Vec<BranchEntry>` and
  `fetch_in_flight: bool`.
- `git::fetch_remote(repo_root)` new helper, run on a blocking thread
  via `async_evt::spawn_fetch` → `Event::FetchDone(r_idx)`.

### PR 5 — Unified delete-worktree modal

**Current state.** Confirming worktree removal chains to a second modal
asking about branch deletion. If the branch isn't fully merged, a
third modal asks about force-delete. Three keystrokes minimum, often
four, and the cascade is surprising.

**Proposed.** Single modal:

```
┌─ Remove worktree for feat/sidebar ───────────────────┐
│                                                       │
│  Worktree path: ~/dev/grove-feat-sidebar              │
│                                                       │
│  ▶ Remove worktree, keep branch                       │
│    Remove worktree and delete branch (git branch -d)  │
│    Remove worktree and force-delete (git branch -D)   │
│                                                       │
│  ⚠ PR #42 is open — force-delete will close it        │
│                                                       │
│  j/k navigate · Enter confirm · Esc cancel            │
└───────────────────────────────────────────────────────┘
```

- Default cursor on "keep branch" (safest).
- `-d` option performs `git branch -d` after worktree removal; if it
  fails with "not fully merged" we show an inline error rather than
  chaining to a third modal ("unmerged; choose force-delete or
  cancel").
- PR warning only renders when option `-D` is selectable and a PR is
  open.

**State changes.**
- Drop `ConfirmDeleteBranch` and `ForceDeleteBranch` modal variants
  (both redundant once the menu covers all outcomes).
- `ConfirmRemoveWorktree` gains a `choice: DeleteChoice` field
  (`KeepBranch` / `Delete` / `ForceDelete`).
- Submit handler runs the selected combination atomically: remove
  worktree, then conditionally branch-delete. Partial failure is
  surfaced as an inline error, the modal stays open.

## Critical review (self)

1. **`grove <path>` adds silently to config.** For a user trying
   grove for the first time, "cd into any repo and run grove" becoming
   permanent may surprise them. Mitigation: show a one-time toast
   ("Added 'grove' to config — press R to remove") on first auto-add.
   If that feels heavy, settle for adding to config only when the user
   interacts beyond initial view.

2. **`grove scan` and nested repos.** Depth-1 default keeps the scan
   fast and avoids monorepo footguns. A user with `~/work/client-a/*`
   and `~/work/client-b/*` runs two scans. Fine.

3. **Auto-fetch and corporate proxies.** Fetch can hang on misconfigured
   networks. Fetches are background and non-blocking for modal
   rendering. The sync-fetch-before-branch should have a timeout
   (e.g., 10s) and fall back to "branch from current HEAD" with a
   warning in the modal.

4. **Unified new-worktree modal and fuzzy ranking.** A naive
   substring match is probably enough (current filter is none). Skip
   a real fuzzy library unless we feel the pain — the list is tens,
   not thousands, of entries.

5. **Unified delete-modal keyboard shortcuts.** Currently
   `y`/`n`/`Enter`/`Esc` map to submit/cancel. Moving to a menu means
   `Enter` replaces `y`. That's a behaviour change for an existing
   modal. Worth calling out in release notes; not worth trying to
   preserve `y` (ambiguous with "yes to which?").

6. **Theme persistence race.** Cycling themes fast could issue
   overlapping saves. Debounce: only persist on app exit, *or* on the
   next tick after the last cycle. Simpler: persist on every cycle and
   accept the write churn (config is small).

7. **`--init` removal is a breaking CLI change.** Users scripting
   `grove --init` in a dotfiles setup will see an error. Acceptable
   given v0.2 shipped the replacement; note in release notes.

8. **Base-branch auto-detect depends on a fetched remote.** If the user
   adds a repo that has a remote but has never been fetched,
   `origin/HEAD` won't resolve. Accept the fallback to `main` and let
   the user override in config if needed.

9. **GitHub auth indicator only fires for GitHub remotes.** GitLab /
   Gitea users see nothing — correct, because we have no PR integration
   for them. When provider support expands (v1.2+), reuse the same
   footer slot.
