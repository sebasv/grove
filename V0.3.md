# Grove v0.3 — Live Git Status

## Goal

Replace the in-memory snapshot of worktrees (taken once at load) with **live status**: every worktree row shows badges for staged / modified / deleted files, ahead/behind counts, and conflicts. Status refreshes automatically when `.git/` changes and on manual `r`.

This is the first phase where grove is async. It introduces the event-loop shape that later phases (PTY readers, PR polling) reuse.

## Acceptance criteria

- Sidebar shows right-aligned status badges per worktree row: `+N` staged, `~N` modified, `-N` deleted, `↑N` ahead, `↓N` behind, `⚠` conflict (in that priority order, leftmost is most urgent). Clean worktrees show `✓`.
- Badges update automatically when files change in any worktree (e.g., `git add`, `git commit`, or an external editor touching a tracked file).
- `r` forces an immediate re-read of all worktrees.
- Startup time is unchanged for small repos; status is read lazily or on a background task so the TUI renders before git status completes.
- The app shells out to `git` exactly once per process (in v0.2's `is_git_repo` validation). All worktree/status reads go through `git2`.
- Modal, navigation, and quit flows still work while background refreshes are in flight.
- Release build clean; 53+ tests pass (existing tests unchanged, new tests for git2 + status derivation).

## Architecture

### Async runtime

```rust
#[tokio::main]
async fn main() -> Result<ExitCode> { ... }
```

Single-threaded (`rt = tokio::runtime::Builder::new_current_thread()`) is fine — we don't need a thread pool, just cooperative multitasking for I/O.

### Event channel

All non-render work flows through one `mpsc::UnboundedSender<Event>` → receiver consumed by the main loop. Variants:

```rust
pub enum Event {
    Input(KeyEvent),
    RepoDirty(RepoId),
    StatusReady(WorktreeId, WorktreeStatus),
    Quit,
}
```

The main loop:

```rust
loop {
    draw(tui, &app);
    tokio::select! {
        maybe = events.recv() => match maybe { ... }
    }
    if app.should_quit { break; }
}
```

### Terminal events

`crossterm::event::EventStream` with the `event-stream` feature. A spawned task forwards:

```rust
tokio::spawn(async move {
    let mut stream = EventStream::new();
    while let Some(Ok(CEvent::Key(key))) = stream.next().await {
        tx.send(Event::Input(key)).ok();
    }
});
```

### Filesystem watcher

One watcher per repo rooted at `<repo_root>/.git/` (non-recursive, because we care about index and refs changes, and both live at that level). `notify::RecommendedWatcher` with a debouncer:

```rust
let (watch_tx, watch_rx) = std::sync::mpsc::channel();
let mut watcher = notify::recommended_watcher(watch_tx)?;
watcher.watch(&dot_git, RecursiveMode::NonRecursive)?;
```

Raw events go through a 150 ms debouncer (bridged to async via `tokio::task::spawn_blocking` running the `watch_rx` stdlib channel) that coalesces bursts and emits one `Event::RepoDirty(repo_id)` per repo per window.

### Status computation

For each worktree, spawn a `tokio::task::spawn_blocking` that calls `git2` and sends `Event::StatusReady`. Blocking pool is appropriate — `git2` is synchronous.

```rust
pub struct WorktreeStatus {
    pub staged: u32,
    pub modified: u32,
    pub deleted: u32,
    pub conflicts: u32,
    pub ahead: u32,
    pub behind: u32,
}
```

`git2` calls used:
- `Repository::open(worktree.path)`
- `repo.statuses(Some(StatusOptions::new().include_untracked(true)))` → iterate, accumulate counts from `entry.status()`.
- `repo.graph_ahead_behind(local_oid, upstream_oid)` for ahead/behind. Upstream resolved via `branch.upstream()`. If no upstream, ahead=0 / behind=0.

## Badge render

32-col sidebar stays; badges right-align and truncate if needed. Budget: left portion shows glyph + branch name (up to ~18 chars); right portion shows up to ~10 chars of badges.

Examples (32 cols):

```
"   ○ main                    ✓ "
"   ● feat/sidebar    +4 ~1 ↑2 "
"   ● fix/deps         ~2    ⚠ "
"   ● wip/zsh             ↑1 ↓2"
```

Priority (leftmost first, drop rightmost if overflow):

```
conflict (⚠) · ci_fail (later) · staged (+N) · modified (~N) · deleted (-N) · ahead (↑N) · behind (↓N) · clean (✓)
```

Colors:
- Green: clean `✓`
- Red: conflict `⚠`
- Yellow: staged / modified
- Dim / white: ahead / behind

`git2` is the heaviest addition (~libgit2 C dep). Acceptable for what we get (full API, zero subprocess).

## Critical review (self)

### What could go wrong

1. **git2 build complexity.** libgit2 links against system OpenSSL/zlib etc. On some systems this fails without `pkg-config`. Mitigation: enable `default-features = false` so we don't pull in optional features (SSH, HTTPS transport). For v0.3 we only need local repo access — no fetch/push — so we can disable net features entirely.

2. **`Repository::open` on every status refresh is wasteful.** Opening the repo allocates. For a 100-worktree user this gets noticeable. Mitigation: cache an open `Repository` per worktree. But `git2::Repository` isn't `Send` — caching across tasks is awkward. Simplest: open fresh each refresh; profile later if needed.

3. **notify false positives.** Git writes many short-lived files in `.git/` (lock files, packed-refs). Some of those aren't meaningful state changes. Debounce window of 150 ms coalesces bursts; we tolerate a small amount of spurious work.

4. **Status computation blocks the UI if not off-thread.** `git2::statuses` can take 100 ms+ on a big repo. We MUST run it on the blocking pool. Mitigation: always spawn_blocking; never call git2 from the main task.

5. **RepoDirty fires while status is already running.** Could queue up work. Mitigation: a per-worktree "in flight" flag prevents concurrent refresh for the same worktree; subsequent dirty events are dropped (the final status read still observes the latest state).

6. **Ahead/behind without upstream.** When `branch.upstream()` returns `NotFound`, set ahead=0, behind=0. Badge omits `↑`/`↓` entirely (don't render as `↑0`).

7. **Badge overflow in narrow terminals.** If sidebar is narrower than expected (user's terminal < 80 cols, for instance), badges may visually smash into the branch name. Mitigation: truncate the branch name with `…` when badges + branch exceed sidebar width. Or reserve a fixed 10-col badge budget and truncate the branch to `sidebar_width - badge_budget - indent`. Go with reserve-and-truncate.

8. **Conflicts counted twice.** A conflicted file shows up in both staged and unstaged status flags. Mitigation: check `Status::CONFLICTED` first and count only as conflict.

9. **The render widget isn't async-aware.** That's fine — render is sync, called from the main loop after a state update. The async work feeds state updates.

10. **Test harness needs real git repos.** Existing v0.2 tests set up temp git repos. Reuse `init_git_repo` from `app.rs` tests. Add helpers to create staged/modified/conflicted scenarios.

11. **Worktrees with submodules.** Status includes submodule changes by default. For v0.3 we leave the default (user probably wants to see submodule dirtiness). Document as a known behavior.

### Alternative designs considered

- **Polling every N seconds instead of notify.** Rejected — bad battery/CPU, missed updates, or delayed UI.
- **Shell out to `git status --porcelain=v2` and parse.** Rejected — we'd swap a parsing problem for a library one; git2 gives structured data and removes subprocess overhead on every watcher tick.
- **`tokio::process::Command` instead of `spawn_blocking` + git2.** Valid alternative but re-introduces subprocess overhead. Skipped.

### Debt / deferrals

- No PR badges (`●`, `◐`, `✓●`, `✗`, `⟳`) — those arrive in v0.8.
- No per-file diff view on the rows — that's the diff viewer in v0.6/0.7.
- `git2::Repository` caching — optimize only if profiling shows it's worthwhile.
- `notify-debouncer-mini` is a conservative pick; `notify-debouncer-full` gives better semantics but at the cost of a tokio channel integration we don't need yet.

### Testing strategy

- **Unit: status derivation** — Create a temp git repo; stage/modify/delete files; assert status counts match. One test per distinct status type.
- **Unit: ahead/behind** — Create repo + upstream; commit on branch; assert ahead/behind resolved correctly; assert "no upstream" returns 0/0 without error.
- **Unit: conflict counted once** — Create conflict scenario (cherry-pick or manual index munging); assert `conflicts=1` and modified=0.
- **Snapshot: sidebar** — Render fixture with pre-seeded statuses; verify badge layout for each scenario.
- **Integration (no notify):** Skip in v0.3 — notify integration tests are fragile on macOS CI; manual smoke test is enough. Reliance on unit tests of the status query and badge render covers the meaningful contract.

## Out of scope for v0.3

- PR / CI badges (v0.8).
- Untracked files in status (they get counted as modified to keep the badge set small). Can split later if users want.
- Worktree list caching or incremental diffs — full re-read each time.
